## 1. Executive summary

Zili is a platform that ingests HTML artifacts generated by LLMs (Claude, ChatGPT, v0, Lovable, Bolt, etc.) and renders them as presentable, navigable, shareable experiences — without any loss of CSS fidelity, layout, or interactivity. Users upload their HTML output once and Zili presents it like a slide deck, a styled document, or any in-between format the artifact implies.

The core thesis: as LLM artifact generation becomes a default workflow, the value increasingly accrues to whoever owns the **render and distribution layer** — not the editor. Zili is positioned to be that layer.

V1 ships as an individual-only product (no teams) with a 3-month free promotion at launch, transitioning to a freemium model with a Pro tier. Brand direction is Editorial Minimal — see the companion brand canvas artifact.

---

## 2. The problem

LLMs now produce visually rich, CSS-driven artifacts as a primary output mode. But the tools people use to *present* (PowerPoint, Google Slides, Keynote) and *share* (PDF, slide hosting) assume a fundamentally different content model — slide objects, text frames, shapes. The HTML-to-deck conversion path is lossy by design:

- Imported HTML loses CSS-driven styling (gradients, custom fonts, grid layouts, animations).
- Page-break semantics break down — multi-section artifacts collapse into single overflowing pages.
- Interactive elements (hover, click reveals, embeds) are stripped.
- Aspect ratios get re-fit awkwardly to 16:9 or letter-size frames.
- Users spend 30–90 minutes manually rebuilding what the LLM produced in 30 seconds — eliminating the productivity gain that motivated using the LLM in the first place.

**The pain is asymmetric:** the better the LLM gets at producing visually distinctive artifacts, the worse the conversion problem becomes.

### The full user category map

Zili has four user categories. V1 must address all of them, even where coverage is light.

| Category | Sub-roles | V1 priority |
|---|---|---|
| **Anonymous visitor** | Marketing site visitor (zili.xyz homepage) | Light — conversion-focused |
| **Anonymous viewer** | Opens a public share link without an account | High — largest user category by volume; primary conversion top-of-funnel |
| **Authenticated customer** | Free tier · Grandfather cohort · Pro tier (post-promo) | Highest — core JTBD lives here |

Each category is addressed in the V1 requirements document with explicit scenarios.

### Primary cohort for the first 100 paying customers (locked)

**Internal-facing PMs** and **independent consultants**. Both share the same underlying behavior profile:

- Generate frequent, high-stakes presentations (weekly+ cadence)
- Use LLMs as a first-draft engine for visual content
- Operate in professional contexts where presentation polish matters
- Have low tolerance for cleanup overhead — time is the binding constraint
- Distribute internally (PMs) or to clients (consultants) — non-public but multi-recipient
- Sophisticated enough not to be put off by a .xyz domain or an indie brand feel

### Core JTBD

> *"When I've used an LLM to draft a deck or document, help me present and share it as-is, without losing the design or wasting time porting it into another tool."*

### Secondary JTBD (V2+)

> *"Let me collaborate on and lightly edit the artifact without leaving the presentation context."*

### Explicitly out of scope for V1

- Team workspaces, shared libraries, role-based access
- Enterprise admin features (SSO, audit logs, retention policies)
- Public-creator workflows (newsletters, course platforms)

| Adjacent product | What it does | Why it doesn't solve this |

| **Gamma, Tome, Beautiful.ai** | Generate decks AI-natively in their own format | Can't ingest external HTML; locked into their templates |

**Zili's wedge:** the *only* tool that says "bring the HTML, we'll present it." The competitive moat is not the editor — it's rendering fidelity, presenter UX, and the share graph.

### Positioning statement

> For people who use LLMs to draft visual artifacts, Zili is the platform that presents and shares those artifacts as living decks and documents — preserving every pixel of the original output, without manual cleanup.

## 5. Product principles

1. **Fidelity above all.** What the LLM produced is what gets shown. No silent re-rendering, no style stripping.
2. **Minimal chrome.** The artifact is the product. UI should disappear during presentation.
3. **Upload, share, present — in that order.** Three taps from raw HTML to a shareable link.
4. **Defer editing.** Editing is a V2 feature, not a V1 trap. The first version wins by being the best renderer, not the worst editor.
5. **Format-agnostic.** Slides, docs, posters, one-pagers — Zili infers the right presentation mode rather than forcing the user to declare it.

## 6. MVP scope (V1)

The goal of V1 is to validate the wedge: **does fidelity-preserving HTML presentation drive enough delight to retain users and earn shares?**

### Must have

- **Upload flow**
  - Drag-and-drop single HTML file (max 10MB)
  - Paste raw HTML string
  - Paste public URL (server-side fetch & render)
- **Rendering**
  - Faithful render via sandboxed iframe (CSP-locked, no parent access)
  - Auto-detect presentation mode (slides vs document) via heuristics:
    - `<section>` tags → slide mode
    - Elements with `.slide`, `.page`, `[data-slide]` → slide mode
    - Tailwind aspect-ratio containers (`aspect-video`, `aspect-[16/9]`) → slide mode
    - Otherwise → document mode (vertical scroll)
  - Manual override toggle in case heuristics misfire
- **Present mode**
  - Fullscreen toggle (browser Fullscreen API + F key)
  - Arrow-key / spacebar navigation (slide mode)
  - Slide counter ("04 / 12") in editorial mono
  - ESC to exit fullscreen
  - Mobile swipe gestures (left/right)
- **Sharing**
  - Generate public shareable URL: `zili.xyz/p/{slug}`
  - Open Graph preview card (matching brand canvas spec)
  - Copy embed code (iframe snippet)
- **Account & library**
  - Email/Google auth
  - Personal library of uploaded artifacts (grid view)
  - Rename, delete, re-upload

### Should have

- Multi-file upload — zip with HTML + CSS + JS + images
- External asset allow-list — preload Tailwind CDN, Google Fonts, common CDNs
- Aspect-ratio detection & scaling — auto-fit to viewer's screen while preserving ratio
- Speaker view — separate window with notes (extract from `aside.notes` or `<!-- notes: ... -->`)

### Could have (V1.5 candidates)

- Password-protected share links
- Custom domain for shared links
- View analytics (open count, time-on-slide)
- Comments on shared decks
- Export to PDF (browser-print pipeline)

### Won't have in V1

- In-platform editor
- Real-time collaboration
- AI-assisted cleanup or generation
- Team workspaces
- Templates / themes

## 10. Design direction

Locked direction: **Editorial Minimal.** See the companion brand canvas artifact (`zili-brand.html`) for the live specification.

### Summary

- **Palette:** Warm paper (`#FAF9F5`), warm ink (`#1A1815`), warm-tinted gray scale, single amber accent (`#D26B2C`).
- **Typography:** Geist (UI/body), Instrument Serif (display/brand moments), Geist Mono (technical accents).
- **Spatial system:** 4px base unit, 4–6px max border radius, generous 32–64px section gaps.
- **Motion:** 150ms ease-out for state, 350ms for transitions, one orchestrated stagger on page load.
- **Wordmark accent:** Amber period after "Zili" — the single repeated brand motif.

### Why Editorial Minimal

- Zili's UI is a *frame* — every pixel of chrome competes with the user's artifact for attention. An editorial aesthetic recedes.
- A warm amber accent harmonizes with users' content (which skews cool — blue/purple gradients dominate LLM output) rather than fighting it.
- A characterful display serif (Instrument Serif) is the brand's single distinctive move — it stops Zili from looking like every other dev-tool SaaS.
- The .xyz domain signals "indie / designed" rather than "enterprise SaaS." Editorial Minimal leans into that without becoming twee.

### Anti-patterns

No gradients. No glassmorphism. No drop shadows beyond 1px. No bright/saturated palettes. No rounded-everything. No emoji in UI chrome. No illustrations or mascots.

## 12. Technical architecture (high level)

### Stack

- **Frontend:** Next.js (App Router) + Tailwind. SSR for shared links (OG previews, SEO).
- **Renderer:** Sandboxed iframe with strict CSP and `sandbox="allow-scripts"` (no `allow-same-origin`).
- **Storage:** Object store (S3 / Cloudflare R2) for raw HTML and uploaded assets. Postgres for metadata, users, sharing.
- **Auth:** Clerk or Supabase Auth — fast to ship, OAuth-ready.
- **Sharing:** Edge-cached shared pages (Cloudflare / Vercel Edge) for sub-100ms loads anywhere.
- **Slide segmentation:** Server-side DOM parse on upload to extract slide boundaries, store an index for fast navigation.

### Critical components

1. **Ingest pipeline** — parse uploaded HTML, sanitize dangerous patterns (inline event handlers, certain script patterns), extract metadata (title, slide count, detected mode), generate thumbnail.
2. **Renderer service** — serves the iframe content with the right CSP headers and asset proxying.
3. **Presenter UI** — minimal chrome wrapper around the iframe, keyboard / gesture handling, fullscreen API.
4. **Share graph** — link generation, OG metadata, optional auth checks.

### Non-functional requirements

- **Performance:** Shared link cold-load < 1s globally (edge cache). Upload-to-render < 5s for files under 5MB.
- **Security:** No script execution outside sandbox. CSP `default-src 'none'` with explicit allow-list. No `allow-same-origin`.
- **Accessibility:** Keyboard navigation for all presenter controls. Screen-reader-friendly chrome. WCAG AA contrast for all UI.
- **Browser support:** Chrome, Safari, Firefox, Edge — latest two major versions each.
- **Mobile:** Responsive presenter for shared links. Mobile upload is V1.5+.

## 13. Admin & operations

For V1 at solo-operator scale, the admin surface is small but non-negotiable. Several capabilities are legal requirements (DMCA, GDPR, account suspension) rather than nice-to-haves.

### Admin roles (V1 vs future)

| Role | V1 scope | Description |
|---|---|---|
| **Operator** | Yes — single role for solo founder | Full access to everything: users, content, billing, security, analytics |
| Support | V2+ | Read-only user lookup, limited intervention (refunds within policy, ticket-driven actions) |
| Moderator | V2+ | Content review, take-downs, abuse queue. No billing access. |
| Billing admin | V2+ | Stripe access, refunds, invoicing. No content access. |

### V1 admin capabilities (must have)

1. **User management** — search/filter users; view profile (plan, artifact count, activity); suspend / reinstate accounts; manually grant or revoke grandfather status (for edge cases); delete user accounts for GDPR.
2. **Content moderation** — review security-scan rejections; handle DMCA take-down requests; review abuse reports queue; force-revoke share links; ban specific patterns.
3. **Billing operations** — issue refunds; grant complimentary Pro subscriptions; apply custom discount codes; view revenue dashboard.
4. **Analytics & observability** — daily/weekly/monthly active users; activation funnel; conversion funnel; cohort retention; top-shared decks; error rates by endpoint; pipeline health.
5. **Audit log** — every admin action is logged with timestamp, actor, target, and reason. Immutable. Reviewable.

### Recommended tooling (V1, lean)

A custom-built admin UI is *not* a V1 priority. Use:

- **Retool** (or Internal, Forest Admin) — for the operator dashboard, built over Postgres + Stripe APIs. Hours to set up, not weeks.
- **Stripe dashboard** — for direct billing operations and revenue analytics.
- **PostHog or Plausible** — for product analytics, funnels, cohort retention.
- **Sentry** — for error monitoring and pipeline health.
- **Direct DB access (e.g. TablePlus)** — for power-user operations and one-off scripts. Behind 2FA.

This stack is < $100/month total and gets you to enterprise-grade ops on day one.

### Trust & safety baseline (V1)

- "Report this content" link on every public shared deck (small, in the footer)
- DMCA contact email and policy page (`zili.xyz/legal/dmca`)
- Terms of service and privacy policy live before launch
- Audit log retained 24 months minimum
- Security-scan rejections retained 90 days for review

## 14. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Fidelity falls short of "feels exactly like the original" | Medium | High | Visual regression QA harness pre-launch; ship narrow but flawless first |
| LLM platforms add native presentation features | High | High | Stay ahead on share graph, collaboration, multi-LLM agnosticism |
| Users want editing in V1 and bounce when they can't get it | Medium | Medium | "Regenerate with prompt" in V2.5; clearly position editing as imminent |
| Security incident (XSS / malicious deck) damages trust | Low | High | Strict sandbox from day one, no `allow-same-origin`, security review process |
| Discoverability / cold start | Medium | High | Watermark on shared decks as passive distribution; 3-month free creates urgency |
| Month-4 paywall causes churn cliff | Medium | Medium | Grandfather cohort discount; cap the cliff with clear comms 30 days out |
