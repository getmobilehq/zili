Operator.

Four surfaces. Active in every phase of every user journey. Built in Retool — designed for clarity.

— Purpose

Per swimlane Insight 03, the Operator lane has no empty cells. Admin tooling is load-bearing infrastructure, not a side concern. The four screens below define the operator's working surface: a dashboard that shows whether anything is wrong, a user detail page for support and intervention, a unified moderation queue for the queues that would otherwise live in three separate tools, and an immutable audit log that protects against disputes. Implementation lives in Retool wired to Postgres, Stripe, and the application API.

Why one queue

Reduces context-switching

Three separate tools = decision fatigue. One unified inbox sorted by severity and SLA = one focused work session. Per swimlane Insight 03.

Context inline

Decision-relevant info on the row

Each item carries enough context to decide without a click. User signals (account age, Google vs email, prior flags) shown next to the reported content.

SLA visibility

Overdue is red, calls itself out

DMCA has the strictest SLA. Visible breach as a red flag in the row header. Operator sees the overdue count in the page header too.

— Screen 04

Audit log. Immutable. Searchable.

Every admin action recorded with timestamp, actor, action, target, and reason. Append-only — entries cannot be edited or deleted, including by operators. Filterable by actor, action type, target, date range. The defence against disputes and the proof of operator discipline.

Immutable

Operators cannot edit or delete entries

Even with full admin privileges. This is what makes the log trustworthy in disputes. The Postgres table has no UPDATE or DELETE grant for operator role.

Every action requires a reason

Structured reason codes

Right-most column shows the reason. Reasons come from a structured list (DMCA, GDPR, abuse, beta access, etc.) — prevents ad-hoc decisions and makes pattern analysis possible.

Filter, filter, filter

Investigation surface

When a dispute arrives, the audit log answers it. Filters by actor, action, target, date range. CSV export for legal review.

— For engineering

Implementation notes.

Per the sprint plan, the admin surface is built in Retool wired to Postgres direct + Stripe API + the application API. The mocks above show the target UX; Retool implements approximations.

Retool, not custom

S8 implementation, weeks not months

Custom admin UI would consume 2-3 sprints. Retool gives 80% of this UX in 1 sprint by composing tables, queries, and forms over Postgres + Stripe APIs.

2FA enforcement

Required from day one

Retool workspace locked to operator email + Google OAuth + 2FA. No exceptions. Single-account compromise is the worst-case admin scenario.

Audit log enforcement

Postgres-level, not application-level

Audit log table has triggers on INSERT only. UPDATE and DELETE grants revoked even from operator role. Even Retool can't modify entries — only insert.

Action wrappers

All actions go through audit-logging wrappers

No raw INSERT/UPDATE/DELETE buttons in Retool. Every action button calls a Postgres function that performs the action AND writes the audit row in one transaction.

Reason codes

Structured enum, not free text

reason_code is a Postgres enum: DMCA_TAKEDOWN, GDPR_DELETE, ABUSE_CONFIRMED, BETA_GRANT, etc. Free-text notes optional. Forces consistency for cross-reporting.

Slack ops channel

Mirror critical events to Slack

Every entry in the audit log with a "critical" severity (suspend, delete, DMCA action, refund > $50) posts to #zili-ops Slack channel in real time. Operator sees changes happening even when not in Retool.
